# Engineer Online — 安全基线规范

> **用途**：定义组织级安全基线，所有模块开发必须遵循。本文档从 `standard/技术架构.md` §10 提取并扩展，作为安全要求的独立权威源。

> **关联文档**：
> - 技术实现细节：`standard/技术架构.md` §10
> - 权限码定义：`design/权限设计.md`
> - 部署安全：`design/部署设计.md` §8

## 1. 认证与授权

### 1.1 JWT 规范

| 项 | 规范 |
|---|---|
| 签发方 | GAC APP 统一认证中心 |
| 算法 | HS256（对称签名） |
| Access Token 有效期 | 2 小时 |
| Refresh Token 有效期 | 7 天 |
| 存储位置（移动端） | Secure Storage（iOS Keychain / Android EncryptedSharedPreferences） |
| 存储位置（H5） | 内存（Pinia store），禁止 localStorage |
| 存储位置（管理后台） | httpOnly Cookie + SameSite=Strict |

### 1.2 网关鉴权

- Gateway 层统一解析 JWT，校验签名和有效期
- 解析成功后注入 Header 到下游服务：
  - `X-User-Id`：用户 ID
  - `X-User-Roles`：角色列表（逗号分隔）
  - `X-User-Permissions`：权限码列表
- 下游服务**禁止**自行解析 JWT，只读 Header

### 1.3 服务间调用

- 内部 Feign 调用携带 `X-Internal-Key`（由 K8s Secret 注入）
- Gateway 拒绝外部请求携带 `X-Internal-Key`（防伪造）
- 服务间调用不经过 Gateway 时，必须校验 `X-Internal-Key`

### 1.4 Permission Code 校验

- 后端使用 `@SaCheckPermission` 注解（见 `design/权限设计.md` §5.1）
- `:own` 范围必须在 Service 层校验资源归属
- 圈子级权限在 Permission Code 之上叠加（先过角色码 → 再过圈子配置）
- **禁止**在 Controller 层硬编码角色判断

## 2. 数据安全

### 2.1 传输加密

| 链路 | 加密方式 |
|---|---|
| 客户端 → Ingress | HTTPS（TLS 1.2+，Ingress TLS 终止） |
| Ingress → 内部服务 | HTTP（K8s 内网，V2 评估 mTLS） |
| 服务 → MySQL | TLS 连接（`useSSL=true`） |
| 服务 → Redis | 密码认证 + TLS（生产环境） |
| 服务 → 外部 API | HTTPS |

### 2.2 存储加密

| 数据类型 | 加密策略 |
|---|---|
| 用户密码 | 不涉及（GAC APP 统一认证） |
| 敏感配置（DB 密码、API Key） | K8s Secret + Vault |
| OSS 文件 | 服务端加密（SSE-KMS） |
| 日志中的敏感字段 | 脱敏后写入（见 §2.3） |

### 2.3 数据脱敏

| 字段类型 | 脱敏规则 | 示例 |
|---|---|---|
| 手机号 | 保留前 3 后 4 | `138****1234` |
| 邮箱 | 保留首字母 + 域名 | `a***@gmail.com` |
| 身份证 | 保留前 3 后 4 | `440***********1234` |
| JWT Token | 仅记录前 8 位 | `eyJhbGci...` |
| IP 地址 | 完整记录（非敏感） | `192.168.1.1` |

**实现方式**：
- 日志输出使用 `@SensitiveField` 注解自动脱敏
- API 响应中的脱敏在 VO 层处理（`@JsonSerialize(using = PhoneMaskSerializer.class)`）
- 管理后台查看完整信息需 `user:view-sensitive:any` 权限

## 3. 输入校验与注入防护

### 3.1 XSS 防护

| 层 | 策略 |
|---|---|
| **入库** | 所有用户输入 HTML 转义（`HtmlUtils.htmlEscape()`） |
| **出库** | 前端使用 `v-text` / `{{ }}` 渲染，**禁止** `v-html`；展示换行用 `white-space: pre-wrap` |
| **富文本** | 不支持。用户输入为纯文本 + 换行（`\n`），所有 HTML 标签全部转义（GR-CONTENT-05） |
| **CSP** | H5 页面设置 `Content-Security-Policy: default-src 'self'; script-src 'self'` |

### 3.2 SQL 注入防护

- **强制**：MyBatis Plus 参数化查询（`#{}`），**禁止**字符串拼接 SQL（`${}`）
- **审计**：CI 扫描 Mapper XML，发现 `${}` 立即报错
- **ORM**：优先使用 LambdaQueryWrapper，避免手写 SQL

### 3.3 CSRF 防护

| 端 | 策略 |
|---|---|
| H5 详情页 | 无状态（JWT in Header），不受 CSRF 影响 |
| 管理后台 | Cookie 认证 + `X-CSRF-Token` 双重校验 |
| API | 所有写操作必须携带 `Authorization` Header |

### 3.4 输入长度与格式校验

所有 API 入参必须在 DTO 层使用 JSR 380 注解校验：

```java
public class QuestionCreateDTO {
    @NotBlank
    @Size(min = 5, max = 400)
    private String title;

    @Size(max = 10000)
    private String content;

    @Size(max = 9)
    private List<@URL String> images;
}
```

**原则**：前端校验是体验优化，后端校验是安全底线。两者必须同时存在。

## 4. 文件上传安全

### 4.1 上传限制

| 类型 | 允许格式 | 大小限制 | 数量限制 |
|---|---|---|---|
| 图片 | jpg / png / gif / webp | ≤ 5MB | 合计 ≤ 9（含视频） |
| 视频 | mp4 | ≤ 5MB | 合计 ≤ 9（含图片） |
| 头像 | jpg / png | ≤ 3MB | 1 |
| 封面图 | jpg / png | ≤ 3MB | 1 |

### 4.2 上传流程安全

1. **前端**：校验文件类型（MIME + 扩展名）和大小
2. **后端**：file-service 签发 OSS 直传 Token（STS 临时凭证，有效期 15 分钟）
3. **OSS**：Bucket Policy 限制上传路径和文件类型
4. **回调**：OSS 上传完成后回调 file-service 校验文件头（Magic Number）
5. **CDN**：图片走 CDN 加速，视频走点播服务

### 4.3 防恶意文件

- 校验文件头 Magic Number（不仅依赖扩展名）
- 图片经 CDN 图片处理服务重新编码（去除 EXIF 和潜在恶意载荷）
- 视频经转码服务处理后才可访问
- 禁止上传可执行文件（.exe / .sh / .bat / .js / .html）

## 5. 接口安全

### 5.1 限流

| 维度 | 策略 | 阈值 |
|---|---|---|
| 单用户单接口 | Sentinel 令牌桶 | 60 次/分钟 |
| 单 IP 全局 | Gateway 层 | 300 次/分钟 |
| 翻译接口 | 单用户 | 10 次/分钟 |
| 投票接口 | 单用户单回答 | 防抖 2 秒 |

超限返回 `429 Too Many Requests`，响应头包含 `Retry-After`。

### 5.2 防重放

- 请求头携带 `X-Timestamp`（Unix 毫秒）和 `X-Nonce`（UUID）
- Gateway 校验：
  - `|当前时间 - X-Timestamp| ≤ 5 分钟`，否则拒绝
  - Redis 记录 Nonce（TTL 5 分钟），重复 Nonce 拒绝
- 仅对写操作（POST / PUT / DELETE）启用

### 5.3 CORS

| 端 | 策略 |
|---|---|
| H5 详情页 | `Access-Control-Allow-Origin` 白名单（GAC APP WebView UA 识别） |
| 管理后台 | IP 白名单 + 域名白名单 |
| 移动端原生 | 不涉及 CORS |

### 5.4 敏感操作二次确认

以下操作需要二次确认（前端弹窗 + 后端校验确认标记）：

- 删除问题/回答
- 取消官方号
- 批量审核驳回
- 清空通知

## 6. 日志与审计

### 6.1 日志规范

| 级别 | 用途 | 示例 |
|---|---|---|
| ERROR | 系统异常、外部依赖失败 | UOP 回调超时 |
| WARN | 业务异常、限流触发 | 用户重复投票 |
| INFO | 关键业务操作 | 问题创建、审核通过 |
| DEBUG | 开发调试（生产关闭） | SQL 执行、缓存命中 |

### 6.2 审计日志

所有管理员操作写入 `t_community_audit_log`（见 `design/数据模型.md` §3.7）：

- 操作人 / 操作类型 / 目标 / 变更前后值 / 时间
- 审计日志**不可删除**（仅归档）
- 保留期限：1 年

### 6.3 日志安全

- **禁止**在日志中输出：JWT Token、密码、完整手机号、完整身份证
- 使用 `@SensitiveField` 注解自动脱敏
- 日志传输走内网 Filebeat → Logstash → ES，不经公网

## 7. 依赖安全

### 7.1 依赖扫描

| 工具 | 频率 | 范围 |
|---|---|---|
| OWASP Dependency-Check | 每次 CI 构建 | Java 后端 |
| npm audit | 每次 CI 构建 | H5 / 管理后台 |
| flutter pub outdated | 每周 | Flutter |
| Snyk（可选） | 每日 | 全栈 |

### 7.2 依赖管理原则

- 使用精确版本号（禁止 `^` / `~` 范围）
- 新增依赖需评估：维护活跃度、Star 数、已知漏洞
- 高危漏洞（CVSS ≥ 7.0）必须 48 小时内修复或降级
- 中危漏洞（CVSS 4.0-6.9）必须 1 周内处理

## 8. 密钥管理

### 8.1 密钥分类

| 密钥类型 | 存储位置 | 轮换周期 |
|---|---|---|
| JWT 签名密钥 | Vault | 90 天 |
| MySQL 密码 | K8s Secret + Vault | 90 天 |
| Redis 密码 | K8s Secret | 90 天 |
| OSS AccessKey | K8s Secret + Vault | 180 天 |
| Google Translate API Key | K8s Secret | 365 天 |
| FCM Server Key | K8s Secret | 365 天 |
| X-Internal-Key | K8s Secret | 30 天 |

### 8.2 密钥使用原则

- **禁止**代码中硬编码任何密钥
- **禁止**将密钥提交到 Git（.gitignore + pre-commit hook 扫描）
- 所有密钥通过环境变量注入（`application-{env}.yml` 引用 `${SECRET_NAME}`）
- 密钥轮换时必须支持双密钥并行（新旧同时有效，过渡期 24 小时）

## 9. 安全事件响应

### 9.1 告警级别

| 级别 | 触发条件 | 响应时间 | 通知渠道 |
|---|---|---|---|
| P0 紧急 | 数据泄露、未授权访问、服务被入侵 | 15 分钟 | 电话 + 企业微信 |
| P1 严重 | 高危漏洞被利用、大规模限流触发 | 1 小时 | 企业微信 |
| P2 一般 | 依赖漏洞发现、异常登录尝试 | 24 小时 | 邮件 |

### 9.2 应急流程

1. **发现** → 安全告警触发或人工上报
2. **止血** → 封禁 IP / 下线服务 / 吊销 Token
3. **定位** → 审计日志 + 链路追踪定位影响范围
4. **修复** → 修补漏洞 + 轮换密钥
5. **复盘** → 48 小时内输出事件报告 + 改进措施

## 10. 安全自检清单

每次发版前，开发者需确认：

- [ ] 所有 API 入参有 DTO 校验注解
- [ ] 写操作有 Permission Code 保护
- [ ] 用户输入已 HTML 转义
- [ ] 无 `v-html`（本项目不使用富文本，全部纯文本 + 换行）
- [ ] 无 `${}` SQL 拼接
- [ ] 文件上传有类型和大小校验
- [ ] 日志无敏感信息明文
- [ ] 新增依赖已通过安全扫描
- [ ] 密钥通过环境变量注入，未硬编码
- [ ] 敏感操作有审计日志

---

## 11. 变更流程

修改安全基线必须：

1. 提交 PR 并标注 `security` 标签
2. 安全负责人 review（不可自审）
3. 更新 `CHANGELOG.md`
4. 通知所有开发者（企业微信群公告）
