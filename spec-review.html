§1 Overview

产品概述

投放管理系统核心价值与关键概念

5 模块 · 38 BR · 47 AC

核心问题

1. 如何按国家、车型、绑车时长精准投放 NPS 问卷？
2. 如何在 App 端以低打扰方式触达车主并控制显示频率？
3. 如何实现 CSI 系统与 App 服务端的数据闭环？

核心目标

1. 支持按多维条件精准筛选目标车主并按比例分配问卷链接。
2. 在 App 端以浮标形式低打扰触达用户，支持灵活的显示频率控制。
3. 打通 CSI 与 App 服务端的数据链路，实现计划下发、数据回流、完成状态更新。

关键特性

🎯 精准分层投放 📊 按比例分配问卷 🔌 CSI 数据闭环 ⏰ 凌晨定时下发 📱 车控页低打扰浮标 🔄 显示频率冷却控制 🌐 WebView H5 容器 🛡 逻辑删除审计

目标用户

运营人员

拥有全部权限：可创建/编辑/启用停用/删除投放计划和投放数据，查看/导入/导出/推送投放数据。

车主用户

已绑定车辆的 GAC APP 用户。在 App 端接收问卷通知、填写并提交 NPS 问卷。

CSI 系统

外部系统，通过 UOP 透传投放计划主数据至 App 服务端，接收投放数据与问卷完成状态。

模块 05.01 P0 Web CMS 平台层

投放计划管理

运营管理后台对 投放管理计划进行全生命周期管理，包括列表查询、新建、编辑、查看、启用/停用、删除；同时接收 CSI 系统通过 UOP 透传的投放计划主数据。

📋 核心功能

  • 投放计划列表查询与筛选
  • 新建/编辑/查看投放计划（右侧抽屉）
  • 启用/停用投放计划
  • 删除投放计划（逻辑删除）
  • 接收 CSI 透传的投放计划主数据

🔗 关联实体 / API

  • Campaign
  • CampaignUrl
  • POST /api/v1/campaigns
  • PUT /api/v1/campaigns/{campaignCode}
  • GET /api/v1/admin/campaigns

🔐 Permission Code

  • campaign:view:any
  • campaign:create:any
  • campaign:update:any
  • campaign:delete:any
  • campaign:enable:any

📐 关键业务规则

  • BR-05.01-01投放计划编码 campaign_code 在系统内全局唯一，不可重复；新建后不可修改。CSI 透传更新时以 campaign_code 作为唯一匹配键。
  • BR-05.01-02同一投放计划内所有 URL 分配比例之和必须等于 100%；保存/透传更新时须校验，否则拒绝。
  • BR-05.01-03计划起止时间仅控制定时下发任务的执行窗口，不影响 App 端入口展示；App 端展示依据每个用户的独立投放时间段。
  • BR-05.01-04投放位置、关闭后隐藏时长、每日下发数量、下发条件、显示图标地址等字段由 App 端提供默认值且可修改。
  • BR-05.01-05投放计划支持启用/停用；停用后不再参与新的定时下发，但已生成的投放记录仍按原规则展示。
  • BR-05.01-06投放计划删除采用逻辑删除（deleted = 1），保留审计轨迹；删除后不再参与下发与查询。
  • BR-05.01-07入口图标图片非必填，上传时限制格式为 jpg/png/webp，单张大小 ≤ 3MB。

✅ 验收标准（AC）

  • AC-05.01-01Given CSI 系统发起有效的投放计划透传请求 When App 服务端接收并校验通过 Then 投放计划被保存或更新。
  • AC-05.01-02Given 运营人员在管理后台新建投放计划，URL 比例之和为 100% When 点击保存 Then 计划保存成功并进入列表。
  • AC-05.01-05Given 计划处于启用状态 When 运营人员点击停用 Then 计划状态变为停用，次日任务不再下发。
  • AC-05.01-06Given URL 比例之和不等于 100% When 点击保存 Then 系统拒绝并提示"分配比例之和必须等于 100%"。
模块 05.02 P0 Web CMS 平台层

投放数据管理

运营管理后台查看与管理 投放管理数据，支持按多维度筛选、导入更新、导出、手动推送至 CSI 以及删除操作。

📋 核心功能

  • 投放数据列表（按下发时间倒序）
  • 多维度筛选（计划名称、手机号、车型、完成状态等）
  • 导入投放数据更新问卷信息
  • 导出投放数据为 Excel/CSV
  • 手动推送投放数据至 CSI

🔗 关联实体 / API

  • CampaignDelivery
  • GET /api/v1/admin/campaigns/deliveries
  • POST /api/v1/admin/campaigns/deliveries/import
  • POST /api/v1/admin/campaigns/deliveries/export
  • POST /api/v1/admin/campaigns/deliveries/push

🔐 Permission Code

  • campaign_delivery:view:any
  • campaign_delivery:import:any
  • campaign_delivery:export:any
  • campaign_delivery:push:any
  • campaign_delivery:delete:any

📐 关键业务规则

  • BR-05.02-01投放数据列表默认按下发时间倒序排列；支持按计划名称/编码、手机号/邮箱、车型、下发时间、完成状态、推送状态筛选。
  • BR-05.02-02列表中的手机号、邮箱等敏感字段须脱敏展示。
  • BR-05.02-03投放时间段由 start_time 和 end_time 两个独立字段定义，与 send_time（下发时间）解耦。
  • BR-05.02-04导入功能根据文件中的 campaign_code + one_id 匹配现有投放记录，更新其问卷 ID、问卷 URL 等字段。
  • BR-05.02-05导出字段与列表字段保持一致；导出文件格式为 Excel 或 CSV，大小限制 ≤ 10MB。
  • BR-05.02-06推送功能仅允许对推送状态为"未推送"或"推送失败"的记录执行。
  • BR-05.02-07推送状态按结果流转：未推送 → 已推送；推送失败可再次手动/自动重试。
  • BR-05.02-08投放记录删除采用逻辑删除（deleted = 1），保留审计轨迹。

✅ 验收标准（AC）

  • AC-05.02-01Given 存在多条投放记录 When 运营人员进入投放数据列表 Then 列表按下发时间倒序展示，手机号/邮箱脱敏。
  • AC-05.02-03Given 运营人员上传合规的导入文件 When 点击导入 Then 系统按 campaign_code + one_id 匹配并更新记录。
  • AC-05.02-06Given 推送失败记录被手动重试 When 推送成功 Then 状态更新为已推送；失败则保持推送失败。
  • AC-05.02-07Given 运营人员上传格式错误的导入文件 When 点击导入 Then 系统拒绝并提示格式错误。
模块 05.03 P0 Backend 平台层

定时下发任务

每日凌晨按投放计划配置筛选符合条件的车主用户，按比例分配问卷链接并生成投放记录；随后将投放数据异步回传至 CSI 系统。

📋 核心功能

  • 凌晨定时执行（xxl-job 分片）
  • 按国家/车型/绑车时间筛选用户
  • 按 URL 比例随机分配问卷链接
  • 生成用户维度投放记录
  • 异步推送投放数据至 CSI

🔗 关联实体 / API

  • Campaign
  • CampaignUrl
  • CampaignDelivery
  • POST /api/v1/campaigns/deliveries/sync

🔐 Permission Code

  • 服务端任务，不绑定 Permission Code

📐 关键业务规则

  • BR-05.03-01定时任务每日凌晨执行一次，默认 02:00，支持 xxl-job 分片执行。
  • BR-05.03-02仅处理启用状态且当前日期在起止时间范围内的投放计划。
  • BR-05.03-03按计划的 country、model 以及 dispatch_condition 筛选用户。
  • BR-05.03-04候选用户按绑车时间由近到远排序，按 daily_dispatch_limit 截断。
  • BR-05.03-05按 URL 比例随机分配问卷链接；同一投放计划对同一用户仅生成一条记录。
  • BR-05.03-06投放记录生成时设置 send_time，start_time/end_time 独立定义投放时间段。
  • BR-05.03-07投放记录生成后异步推送至 CSI；失败时标记 PUSH_FAILED 并支持重试。
  • BR-05.03-08若某计划当日无符合条件的用户，则不生成任何记录，任务正常结束。

✅ 验收标准（AC）

  • AC-05.03-01Given 到达配置的执行时间 When xxl-job 触发定时任务 Then 任务正常启动并按计划执行。
  • AC-05.03-03Given 计划配置了下发条件 When 任务筛选用户 Then 仅绑车时间满足条件的用户进入候选池。
  • AC-05.03-05Given 截断后用户已确定 When 分配问卷链接 Then 按 URL 比例分配且同一计划对同一用户仅生成一条记录。
  • AC-05.03-07Given 投放记录已生成 When 异步推送至 CSI Then 成功则标记 PUSHED，失败则标记 PUSH_FAILED。
模块 05.04 P0 Flutter 业务层

App 问卷入口

在 GAC APP 车控页右下角以浮标形式展示 NPS 问卷入口；支持用户关闭入口，并按冷却规则控制再次显示。

📋 核心功能

  • 查询用户有效问卷数据
  • 本地 coolOff 显示规则判断
  • 浮标展示与关闭交互
  • 持久化各位置下次可显示时间

🔗 关联实体 / API

  • CampaignDelivery
  • CampaignSurveyVO
  • GET /api/v1/campaigns/user/surveys

🔐 Permission Code

  • 车主身份隐式校验（通过 one_id 隔离）

📐 关键业务规则

  • BR-05.04-01客户端在获取车辆信息完成后调用问卷查询接口，可返回多条不同位置的投放记录。
  • BR-05.04-02问卷数据返回规则：计划启用、当前用户存在已下发且未完成的问卷、当前时间在用户投放时间段内。
  • BR-05.04-03问卷入口显示规则：服务端返回了问卷数据、本地 coolOff 时间已到期、该位置未处于永久关闭状态。
  • BR-05.04-04客户端以展示位置为维度，在本地持久化存储各位置的下次可显示时间戳（APP_AD_COOL_OFF_MAP）。
  • BR-05.04-05用户关闭浮标时根据接口返回的 cool_off 计算下次可显示时间；cool_off = 0 表示永久不再显示。
  • BR-05.04-06用户完成问卷或投放时间段过期后，对应投放记录不再返回，浮标自动隐藏。
  • BR-05.04-07同一位置存在多条有效投放记录时，关闭当前不影响其他记录。

✅ 验收标准（AC）

  • AC-05.04-01Given 用户进入车控页 When 调用问卷查询接口 Then 返回当前用户有效的问卷数据列表。
  • AC-05.04-03Given 客户端已获得问卷数据且 coolOff 已到期 When 进入车控页 Then 在指定位置展示 NPS 浮标。
  • AC-05.04-04Given 用户关闭浮标 When 客户端处理关闭事件 Then 按 cool_off 计算并持久化下次可显示时间。
  • AC-05.04-08Given cool_off 配置为关闭后不再显示 When 用户关闭浮标 Then 该位置后续不再展示该计划入口。
模块 05.05 P0 H5 WebView 业务层

问卷页

App 以 WebView 容器加载 CSI 系统提供的问卷 H5；用户填写后 H5 调用 App JSBridge，由 App 上报完成状态并隐藏浮标。

📋 核心功能

  • WebView 加载 CSI H5 问卷页
  • H5 内部 Submit 触发 JSBridge 回调
  • App 调用完成接口上报状态
  • 幂等处理重复完成上报

🔗 关联实体 / API

  • CampaignDelivery
  • POST /api/v1/campaigns/user/surveys/complete

🔐 Permission Code

  • 车主身份隐式校验

📐 关键业务规则

  • BR-05.05-01问卷页由 CSI 系统提供的 H5 URL 渲染，App 仅做 WebView 容器。
  • BR-05.05-02提交按钮位于 H5 内部；提交时 H5 须调用 App 原生 JSBridge 方法 npsSurveyCompleted。
  • BR-05.05-03App 收到 JSBridge 回调后调用服务端完成接口；成功后将对应投放记录标记为已完成并隐藏浮标。
  • BR-05.05-04完成接口需幂等处理；已完成的记录再次调用时返回成功，不重复更新。
  • BR-05.05-05WebView 加载 H5 失败时显示重试页面，用户可点击重试或返回车控页。
  • BR-05.05-06H5 与 App 的通信须兼容 WebView 安全区、返回手势及主题同步规范。

✅ 验收标准（AC）

  • AC-05.05-01Given 用户从浮标进入问卷页 When WebView 加载 CSI H5 URL Then 页面正确渲染问卷内容。
  • AC-05.05-03Given App 收到 JSBridge 完成回调 When 调用完成接口成功 Then 服务端将投放记录标记为已完成。
  • AC-05.05-04Given 用户已完成问卷 When H5 再次触发完成回调 Then 服务端幂等返回成功。
  • AC-05.05-05Given WebView 加载 H5 失败 When 页面显示重试页 Then 用户可点击重试或返回车控页。

⚡ 性能与安全
  • ·GR-AUTH-01 · 管理后台使用 httpOnly Cookie + SameSite=Strict 认证；App 客户端使用 JWT Bearer Token。
  • ·GR-AUTH-02 · 服务间调用须携带 X-Internal-Key，Gateway 拒绝外部请求携带该 Header。
  • ·GR-PERM-01 · 后端使用 @SaCheckPermission 注解校验 Permission Code，禁止在 Controller 硬编码角色判断。
  • ·GR-PERM-02 · 车主端接口只能操作当前登录用户自己的投放记录。
  • ·GR-PAGE-01 · 管理后台使用传统 page/pageSize 分页，默认 20 条/页，最大 100 条/页。
  • ·GR-PAGE-02 · App 客户端列表接口如需分页，使用 cursor 分页。
  • ·GR-NOTIFY-01 · 投放数据批量推送失败时，向运营人员发送站内通知或邮件。
  • ·GR-CONTENT-01 · 计划名称、计划编码等文本字段须做 XSS 转义与长度校验。
  • ·GR-CONTENT-02 · 入口图标图片限制 jpg/png/webp，单张 ≤ 3MB。
  • ·GR-ERROR-01 · 统一返回 {code, message, data, traceId, timestamp} 结构；业务错误码见 design/错误码.md。
  • ·GR-AUDIT-01 · 所有运营人员对投放计划与投放数据的写操作写入审计日志（操作人、类型、资源 ID、变更前后值、时间、IP），保留 1 年。

⚡ 性能目标

  • ·管理后台列表加载 ≤ 500ms (P95)
  • ·用户问卷查询接口 ≤ 200ms (P95)
  • ·定时下发任务 1 小时内完成
  • ·投放数据导出 ≤ 30s（10 万条以内）

🛡 安全要求

  • ·手机号/邮箱在管理后台列表中脱敏展示
  • ·导入/导出文件格式校验与大小限制（≤ 10MB）
  • ·CSI 透传接口仅允许服务间调用，须校验 X-Internal-Key

部署要点

  • ·campaign-service 作为独立微服务部署于 K8s 集群，通过 Istio Ingress Gateway 暴露接口
  • ·管理后台构建为 Vue 3 SPA，部署至 Nginx 静态资源服务器
  • ·App 端问卷入口由 GAC APP 统一发布，WebView 内的 H5 由 CSI 系统独立发布
  • ·xxl-job 调度器独立部署，凌晨定时任务分片广播执行
  • ·Redis 集群模式 + MySQL 主从架构保障高可用
1

App 端入口用 Flutter 原生还是 H5？

Flutter 原生
车控页为 Flutter 原生页面，浮标叠加为 light-weight widget；问卷内容由 CSI 独立 H5 提供，App 仅做 WebView 容器。两者各取所长。
2

投放数据推送失败如何处理？

PUSH_FAILED + 手动重试
标记推送失败状态并记录日志，支持运营人员在管理后台手动重试推送，同时支持定时自动重试。
3

删除是物理删除还是逻辑删除？

逻辑删除
投放计划与投放记录均采用逻辑删除（deleted = 1），保留审计轨迹用于合规与问题追溯。
4

问卷提交由谁触发完成回调？

H5 JSBridge → App → 服务端
CSI H5 内部 Submit 调用 npsSurveyCompleted JSBridge 方法，App 接收后调服务端完成接口上报状态。

§9 AI Coding Guide

AI 开发指南

基于 Skill 的 Spec Coding 开发流程（Kiro Skills + Claude Code Skills）

什么是 Spec Coding

以 Spec 文档为单一真相源（SSOT）的 AI 辅助开发模式。BR/EX/AC 编号体系形成"需求 → 测试 → 任务 → API"全链路追溯，AI 严格按文档生成代码并强制输出自检清单，确保实现可验证、可追溯。

人在回路（Human-in-the-Loop）

每个 Skill 遵循"先确认方案，再生成代码"原则。AI 输出实施方案表格供开发者确认，避免方向错误导致 Token 浪费与返工。
