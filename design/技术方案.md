## 3. 服务划分

NPS 问卷投放模块作为独立业务域，建议新增 `campaign-service` 微服务（或在现有 `community-service` 中扩展，视团队服务划分策略而定）。

| 服务 | 职责 | 说明 |
|------|------|------|
| `campaign-service`（新增） | 投放计划管理、定时下发、投放数据管理、用户问卷查询、完成状态更新 | 独立部署，便于未来扩展 |
| `user-service`（复用） | 用户信息、oneId 查询 | 已有服务 |
| `vehicle-service`（复用） | 车辆信息、绑车时间查询 | 已有服务 |
| `file-service`（复用） | 入口图标上传 OSS | 已有服务 |
| CSI / UOP | 计划来源、数据回流目标 | 外部系统 |

## 5. 定时任务设计

### 5.1 任务信息

| 项 | 值 |
|----|-----|
| 任务名称 | npsDailyDispatchJob |
| 执行器 | xxl-job |
| Cron | `0 0 2 * * ?`（默认每日 02:00） |
| 路由策略 | 分片广播 |
| 失败处理 | 记录日志，告警 |

### 5.2 任务流程

1. 查询启用且处于起止时间内的计划。
2. 对每个计划，调用用户/车辆服务获取候选用户。
3. 排除已下发过的用户。
4. 按绑车时间排序并截断。
5. 按 URL 比例分配问卷链接。
6. 批量写入投放数据表。
7. 异步推送至 CSI。

## 6. 数据同步策略

### 6.1 CSI → App 服务端

- 方式：UOP HTTPS 透传
- 场景：计划创建/更新
- 鉴权：`X-Internal-Key`

### 6.2 App 服务端 → CSI

- 方式：UOP HTTPS 同步 + 失败重试
- 场景：投放数据同步、完成状态同步（可选）
- 失败处理：标记 `PUSH_FAILED`，进入重试队列

## 7. 缓存策略

| 缓存 | 类型 | TTL | 说明 |
|------|------|-----|------|
| `nps:plan:{campaign_code}` | Redis String | 1h | 投放计划详情 |
| `nps:user:surveys:{one_id}` | Redis String | 10m | 用户有效问卷列表 |
| `APP_AD_COOL_OFF_MAP` | App 本地 SharedPreferences | 持久 | 各位置下次可显示时间 |

---

## 8. 安全要点

- CSI 透传接口仅允许服务间调用，校验 `X-Internal-Key`。
- 管理后台写操作校验 Permission Code。
- 车主端接口通过 `one_id` 隔离，只能操作自己的投放记录。
- 敏感字段（手机号、邮箱）脱敏展示与存储。
- 文件上传限制类型与大小。
